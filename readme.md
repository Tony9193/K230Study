## 项目简介

本项目基于立创K230/CanMV开发板，演示如何用摄像头+触摸屏+AI手势/色块追踪，实现一个“既能触摸也能隔空操作”的简易计算器。适合零嵌入式基础、仅有Python基础的初学者学习K230软硬件、AI推理、UI绘制、输入处理等核心技能。

---

## 我的需求分析

- **目标**：
  - 用K230做一个能用触摸屏和AI手势/色块隔空操作的计算器。
  - 要求代码有详细注释，适合初学者阅读。
  - 体验优先，能自动降级，稳定不卡死。
- **输入方式**：
  - 触摸屏（优先，最稳定）
  - AI手势（优先，体验好时用）
  - 色块追踪（AI不可用时自动回退）
- **输出方式**：
  - LCD屏幕，底图为摄像头画面，叠加半透明UI按钮和结果。
- **易用性**：
  - 代码结构清晰，注释详细，适合零K230基础的Python初学者。
  - 运行稳定，内存自动保护，调试信息丰富。

## 整体技术原理

1. **摄像头采集**：实时获取环境画面，作为底图。
2. **触摸/AI/色块输入**：
   - 触摸屏直接点按钮。
   - AI模型识别手掌/关键点/手势，定位指尖作为“鼠标”。
   - 色块追踪识别特定颜色的“手指”，隔空点按钮。
3. **UI绘制**：用OSD叠加层画出半透明按钮、表达式、结果、光标等。
4. **主循环**：每帧采集输入→识别→绘制UI→显示，保证流畅交互。
5. **内存与稳定性保护**：周期性GC、内存池收缩、低内存自动降级，防止卡死。

## 核心功能与实现流程

1. **初始化摄像头与显示屏**
   - 设置分辨率、像素格式，保证UI和输入坐标一致。
2. **初始化AI模型与色块追踪参数**
   - 优先用AI手势，失败自动降级色块追踪。

   - 采集摄像头画面

4. **输入处理优先级**
   - 触摸 > AI手势 > 色块追踪
5. **稳定性机制**
   - 自动GC、内存池收缩、低内存det-only、异常降级保护

## AI手势识别原理简述

- **手掌检测模型（hand_det.kmodel）**：
  - 输入摄像头画面，输出手掌框坐标
- **关键点检测模型（handkp_det.kmodel）**：
  - 输入手掌区域，输出21个关键点（含指尖）
- **手势分类**：
  - 通过关键点角度关系，区分one/gun/yeah等手势
- **指针追踪**：
  - 取食指指尖坐标，平滑后作为“鼠标”
- **降级保护**：
  - 内存不足/推理失败时自动切回手掌框中心

---

## 色块追踪原理简述

- **find_blobs**：
  - 在LAB颜色空间下找出指定颜色的连通区域
- **手指识别**：
  - 取最大色块中心作为“手指”
- **长按判定**：
  - 必须稳定停留1秒才算点击，防止误触

## 运行与调试教程

1. **准备工作**
   - 烧录CanMV固件，确保摄像头/触摸屏可用
   - 上传AI模型到板载kmodel目录
   - 上传主程序到开发板
2. **运行脚本**
   - 用CanMV IDE或串口终端运行LCKFB_study(finger_Calculater).py
   - 屏幕显示摄像头画面+半透明计算器UI
3. **操作方式**
   - 直接用手指触摸屏幕按钮
   - 或用红色物体/手指在摄像头前隔空点按钮
   - 若AI模型可用，支持one/gun/yeah等手势指针
4. **调试与排错**
   - 屏幕/串口有详细调试信息（帧率、内存、步骤、AI状态等）
   - 若卡死/无响应，重启板子即可
   - 若AI不可用，自动降级为色块追踪

## 常见问题与FAQ

- **Q: PC上运行报一堆未定义/无法导入？**
  - A: 这些库和类都是K230板子专用，PC上无法直接运行，需在板子上测试。
- **Q: 触摸/AI/色块都没反应？**
  - A: 检查硬件连接、固件版本、模型路径，重启板子。
- **Q: 运行一段时间后卡死？**
  - A: 已加内存保护和降级机制，若仍卡死请升级固件或减少AI推理分辨率。
- **Q: 如何调整色块追踪灵敏度？**
  - A: 修改finger_threshold参数，或用不同颜色物体测试。

### 🔴 已修复 - 真实 Bug / 崩溃风险

| # | 问题描述 | 风险等级 | 修复方案 |
|---|----------|----------|----------|
| 1 | `calculate()` 只捕获 3 种异常，`eval('9**9**9')` 会抛 `OverflowError` 导致程序崩溃 | **高** | 改为 `except Exception` 统一捕获 |
| 2 | `SimpleAi2d` 无 `deinit()`，`nn.ai2d()` / `output_tensor` / `builder` 持有的 NPU 内存永不释放，是内存碎片积累和"运行20秒后卡死"的根因之一 | **高** | 新增 `SimpleAi2d.deinit()`，并在 `SimpleAIBase.deinit()` 中调用 |
| 3 | AI推理异常时 `del ai_img` 被跳过，帧缓冲区资源泄漏 | **中** | 用 `try/finally` 包裹推理块，保证 `ai_img` 必被释放 |
| 4 | 初始化时的临时 `clear_img`（800×480×4 ≈ 1.5MB）用完后没有 `del`，等待 GC 期间持续占用堆内存 | **中** | `Display.show_image` 后立即 `del clear_img` |

### 🟡 已修复 - 逻辑问题 / 隐患

| # | 问题描述 | 修复方案 |
|---|----------|----------|

### ✅ 审查通过 - 做得正确的部分

- 降级逻辑（AI → 色块）完整，有 `try/except` 保护，不会硬死
- `finally` 块中的资源清理（`sensor.stop`, `Display.deinit`）写法规范
- `os.exitpoint()` 保留了 CanMV IDE 优雅停止能力
- `FORCE_GC_INTERVAL_FRAMES` + `SHRINK_POOL_INTERVAL_FRAMES` 的周期 GC 策略思路正确
- `smooth_point` 的自适应平滑（大位移提高 alpha）是个好技巧

### K230 专项优化小技巧

1. **ai2d builder 重建开销**：`crop_and_resize` 每帧都重建 `builder`，这在 K230 上有一定CPU开销。若手掌框变化不大（连续帧间位移 < 20px），可复用上一帧的 builder 跳过重建。

2. **KPU 内存池碎片**：K230 的 KPU 内存是固定池，频繁分配/释放小张量会产生碎片。`nn.shrink_memory_pool()` 能合并碎片，建议每 60~120 帧调用一次（现有代码已做到）。

3. **双通道 sensor 内存**：AI 模式下 chn0（800×480 RGB888）+ chn2（640×360 RGBP888）同时采集，合计约 3MB 帧缓冲。若内存告急，可把 chn2 降至 320×240。

4. **to_numpy_ref() vs to_numpy()**：使用 `to_numpy_ref()` 返回的是零拷贝引用，不分配新内存，是正确选择，比 `to_numpy()` 省约一帧大小的内存。
